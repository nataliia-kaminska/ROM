# РОЗДІЛ 2. ПОСТАНОВКА ЗАДАЧІ ТА СПЕЦИФІКАЦІЯ ВИМОГ ДО СИСТЕМИ ПОШУКУ ТА РЕКОМЕНДАЦІЇ НАУКОВИХ ГРАНТІВ

## 2.1. Постановка задачі та характеристика об’єкта проєктування

Метою кваліфікаційної роботи є проєктування та розробка інтелектуальної веб-системи для автоматизованого пошуку, агрегації та персоналізованої рекомендації наукових грантів і можливостей фінансування для дослідників.

Для досягнення поставленої мети необхідно виконати такі основні завдання:

1.  Проаналізувати вимоги та спроєктувати масштабовану клієнт-серверну архітектуру, здатну обробляти та інтегрувати великі обсяги даних з різнорідних зовнішніх джерел.
2.  Розробити підсистему збору даних (ingestion) для автоматичного завантаження, парсингу та нормалізації інформації про гранти та наукові публікації з таких джерел, як Grants.gov, OpenAlex та ORCID.
3.  Реалізувати підсистему управління профілями користувачів, що дозволяє дослідникам створювати деталізовані профілі, включаючи інформацію про їхні наукові інтереси, публікації та професійний досвід.
4.  Спроєктувати та розробити рекомендаційний механізм, який на основі аналізу профілю користувача та семантичної близькості наукових робіт пропонує найбільш релевантні можливості фінансування.
5.  Розробити інтерактивний графічний інтерфейс для візуалізації рекомендованих грантів, розширеного пошуку та управління профілем користувача.

**Об’єктом проєктування** є процес взаємодії дослідника з консолідованою інформацією про можливості фінансування та наукові дані у середовищі веб-браузера.

Відповідно до специфіки роботи системи, **вхідними даними** для розробленого програмного забезпечення є:

1.  **Контентні (зовнішні) дані**:
    *   Інформація про гранти та можливості фінансування з порталу Grants.gov.
    *   Метадані наукових публікацій, профілі авторів та інформація про наукові установи з бази даних OpenAlex.
    *   Верифіковані профілі дослідників та списки їхніх публікацій з ORCID.
2.  **Користувацькі (внутрішні) дані**:
    *   Дані для реєстрації та автентифікації (наприклад, через ORCID OAuth).
    *   Інформація, яку користувач вносить у свій профіль: наукові інтереси, ключові слова, біографія.
    *   Параметри пошукових запитів та фільтрів.

**Результатом роботи** програмного забезпечення (вихідними даними) є:

1.  **Динамічний користувацький інтерфейс**:
    *   Персоналізована стрічка рекомендованих грантів.
    *   Сторінка з результатами розширеного пошуку та фільтрації.
    *   Інформативний профіль користувача, що агрегує його дані.
2.  **Повідомлення та сповіщення**: Автоматизовані email-повідомлення про нові релевантні гранти, що відповідають критеріям користувача.
3.  **Аналітичні дані**: Збережена історія рекомендацій та взаємодії користувача з системою для подальшого покращення алгоритмів.

## 2.2. Обґрунтування вибору технологічних засобів та інструментів реалізації

Вибір технологічного стека зумовлений вимогами до асинхронної обробки даних, інтеграції з численними зовнішніми сервісами та необхідністю побудови надійної, масштабованої системи.

### 2.2.1. Серверна частина (Back-end)

*   **Мова та фреймворк**: **Python 3.11** та **FastAPI**. FastAPI обрано через його високу продуктивність, що досягається завдяки асинхронному веб-серверу **Uvicorn** (реалізація ASGI). Нативна підтримка `async/await` є критичною для неблокуючої обробки HTTP-запитів до зовнішніх API (OpenAlex, Grants.gov). Валідація вхідних даних та серіалізація відповідей реалізована декларативно за допомогою моделей **Pydantic**, що також слугують основою для автоматичної генерації OpenAPI-специфікації.

*   **Доступ до даних та СУБД**: В якості СУБД використовується **PostgreSQL**. Взаємодія з нею реалізована через **SQLAlchemy 2.0** в асинхронному режимі з використанням драйвера `asyncpg`. Це дозволяє виконувати запити до бази даних, не блокуючи event loop. Для управління міграціями схеми бази даних застосовується **Alembic**.

*   **Архітектура**: Серверна частина спроєктована за принципами **Чистої архітектури (Clean Architecture)**, що забезпечує слабку зв'язність компонентів та високий рівень тестування. Структура проєкту чітко відображає цей підхід:
    *   `app/domain`: Містить моделі SQLAlchemy (Entities) та бізнес-логіку, незалежну від фреймворку.
    *   `app/application`: Визначає сценарії використання (Use Cases) та абстрактні інтерфейси (порти), наприклад, `UnitOfWork` та `OpportunitiesRepository`.
    *   `app/infrastructure`: Надає конкретні реалізації інтерфейсів з `app/application`. Наприклад, `SqlAlchemyUnitOfWork` та реалізації сховищ, що працюють з `AsyncSession` SQLAlchemy. Тут же знаходяться конектори до зовнішніх API.
    *   `app/api`: Шар FastAPI, що містить ендпоінти. Використовує механізм `Depends` для ін'єкції залежностей (наприклад, для отримання `UnitOfWork`), обробляє HTTP-запити та повертає DTO (Data Transfer Objects) на базі Pydantic.

### 2.2.2. Автентифікація та авторизація

Система підтримує два механізми автентифікації:
1.  **OAuth 2.0 з ORCID**: Для основної автентифікації користувачів-дослідників, що дозволяє отримати верифікований ORCID ID та список публікацій.
2.  **JWT-токени**: Для доступу до захищених ендпоінтів API. Після успішної автентифікації генерується access-токен.

### 2.2.3. Фонові задачі та кешування

*   **Фонові задачі**: Для виконання довготривалих операцій, таких як періодичний збір даних з Grants.gov або масовий імпорт публікацій, використовується фреймворк **Celery**. В якості брокера повідомлень для Celery виступає **Redis**.
*   **Кешування**: **Redis** також використовується для кешування результатів запитів до зовнішніх API та інших даних, що рідко змінюються, з метою зменшення затримок та навантаження на систему.

### 2.2.4. Клієнтська частина (Front-end)

Інтерфейс користувача розроблено як **Single Page Application (SPA)**.
*   **Інструменти збірки**: Проєкт збирається за допомогою **Vite**, що забезпечує швидку розробку завдяки механізму Hot Module Replacement (HMR).
*   **Мова**: **TypeScript** використовується для забезпечення типізації, що підвищує надійність коду та покращує досвід розробки.
*   **Взаємодія з Back-end**: Клієнтська частина взаємодіє з серверною через **RESTful API**, надсилаючи та отримуючи дані у форматі JSON.

### 2.2.5. Розширені технічні рішення

Для підвищення надійності, продуктивності та зручності розгортання системи, впроваджено наступні рішення:

*   **Сповіщення в реальному часі**: Для миттєвого інформування користувачів про появу нових релевантних грантів, система використовує протокол **WebSocket**. FastAPI надає нативну підтримку для встановлення постійного двостороннього з'єднання між клієнтом та сервером. Коли фонове завдання (Celery) знаходить новий грант, що відповідає профілю активного користувача, сервер надсилає повідомлення через WebSocket, яке миттєво відображається в інтерфейсі без необхідності оновлення сторінки.

*   **Повнотекстовий пошук**: Для забезпечення швидкого та релевантного повнотекстового пошуку по описах грантів та наукових публікацій, система інтегрована з пошуковим рушієм **Elasticsearch**. Замість неефективних `LIKE` запитів до реляційної бази даних, дані індексуються в Elasticsearch. Це дозволяє реалізувати складні пошукові запити, включаючи морфологічний пошук, ранжування результатів за релевантністю (на основі алгоритму TF-IDF/BM25) та застосування складних фільтрів з високою продуктивністю.

*   **Контейнеризація та розгортання**: Для забезпечення портативності та відтворюваності середовища, весь застосунок контейнеризовано за допомогою **Docker**. Створено окремі образи для FastAPI-сервера, Celery-воркера та фронтенд-частини. Для управління всіма компонентами системи (включаючи PostgreSQL, Redis та Elasticsearch) використовується **Docker Compose**. Файл `docker-compose.yml` описує всі сервіси, їхні залежності та мережеву взаємодію, що дозволяє запустити весь стек застосунку однією командою, гарантуючи ідентичність середовища розробки, тестування та продуктивного використання.

## 2.3. Специфікація вимог

Цей підрозділ містить визначення функціональних та нефункціональних вимог до розроблюваної системи пошуку та рекомендації наукових грантів.

### 2.3.1. Класи користувачів та їх характеристики

Система передбачає взаємодію з трьома основними ролями (класами) користувачів:

1.  **Гість (Guest)**: Неавторизований користувач. Може переглядати загальнодоступну інформацію, таку як опис системи та приклади грантів, але не має доступу до персоналізованих функцій.
2.  **Дослідник (Researcher)**: Авторизований користувач, який пройшов автентифікацію через ORCID. Це ключова роль в системі. Дослідник має персональний профіль, отримує рекомендації, може використовувати розширений пошук та налаштовувати сповіщення.
3.  **Адміністратор (Administrator)**: Користувач з розширеними правами. Відповідає за моніторинг стану системи, управління процесами збору даних (ingestion), перегляд системної аналітики та управління користувачами.

### 2.3.2. Характеристики системи та функціональні вимоги

Функціональні вимоги організовані за ключовими модулями системи.

**Характеристика 1: Підсистема автентифікації та управління профілем**
*   **Опис і пріоритет**: Високий. Забезпечення безпечного доступу та створення деталізованого профілю користувача для точних рекомендацій.
*   **Функціональні вимоги**:
    *   **REQ-1.1**: Система повинна реалізувати автентифікацію користувачів через протокол OAuth 2.0 з використанням зовнішнього провайдера ORCID.
    *   **REQ-1.2**: Після першої автентифікації система повинна автоматично створювати профіль користувача та імпортувати базову інформацію (ПІБ, місце роботи) та список наукових публікацій з його профілю ORCID.
    *   **REQ-1.3**: Дослідник повинен мати можливість доповнювати та редагувати свій профіль, вказуючи ключові наукові інтереси у вигляді тегів, розгорнуту біографію та сфери компетенції.
    *   **REQ-1.4**: Система повинна надавати можливість ручної синхронізації списку публікацій з профілем ORCID для отримання останніх оновлень.
    *   **REQ-1.5**: Профіль користувача повинен відображати статистику його активності в системі: кількість переглянутих грантів, кількість поданих заявок (якщо відстежується), дата останньої активності.

**Характеристика 2: Підсистема збору та обробки даних (Ingestion)**
*   **Опис і пріоритет**: Високий. Забезпечення системи актуальними та повними даними про гранти та наукові публікації.
*   **Функціональні вимоги**:
    *   **REQ-2.1**: Система повинна виконувати щоденне автоматичне завантаження нових даних про гранти з офіційного API Grants.gov.
    *   **REQ-2.2**: Система повинна інтегруватися з API OpenAlex для періодичного збагачення даних про наукові публікації (отримання анотацій, ключових слів, цитувань, інформації про співавторів).
    *   **REQ-2.3**: Всі отримані дані повинні проходити процес нормалізації: очищення HTML-тегів, приведення дат до єдиного формату, валідація числових значень.
    *   **REQ-2.4**: Система повинна вести детальний журнал (лог) кожної сесії збору даних, фіксуючи джерело, кількість отриманих записів та можливі помилки.
    *   **REQ-2.5**: Реалізувати механізм ідентифікації та об'єднання дублікатів грантів, що можуть надходити з різних джерел або повторно.

**Характеристика 3: Підсистема рекомендацій та пошуку**
*   **Опис і пріоритет**: Високий. Ядро системи, що надає користувачу релевантну інформацію.
*   **Функціональні вимоги**:
    *   **REQ-3.1**: Система повинна генерувати персоналізований список рекомендованих грантів, використовуючи алгоритм на основі векторного представлення текстів (наприклад, TF-IDF або Doc2Vec) для порівняння анотацій публікацій користувача з описами грантів.
    *   **REQ-3.2**: Дослідник повинен мати доступ до сторінки розширеного пошуку з можливістю фільтрації грантів за: діапазоном дат, розміром фінансування, науковою галуззю (категорією), ключовими словами.
    *   **REQ-3.3**: Пошук повинен бути реалізований з використанням повнотекстового індексу в Elasticsearch для забезпечення високої швидкості та підтримки морфологічного пошуку.
    *   **REQ-3.4**: Система повинна підтримувати можливість збереження наборів пошукових фільтрів для їх швидкого повторного застосування.
    *   **REQ-3.5**: Користувач повинен мати можливість позначити грант як "нецікавий", щоб виключити його з майбутніх рекомендацій та покращити точність алгоритму.
    *   **REQ-3.6**: На сторінці кожного гранту повинні відображатися схожі на нього гранти.

**Характеристика 4: Підсистема сповіщень**
*   **Опис і пріоритет**: Середній. Проактивне інформування користувачів.
*   **Функціональні вимоги**:
    *   **REQ-4.1**: Система повинна надсилати щотижневі email-дайджести з підбіркою найбільш релевантних нових грантів.
    *   **REQ-4.2**: Система повинна надсилати миттєві сповіщення в інтерфейсі через WebSocket, коли з'являється грант, що має високий ступінь відповідності (>90%) профілю користувача.
    *   **REQ-4.3**: Користувач повинен мати можливість в налаштуваннях профілю вмикати/вимикати різні типи сповіщень (email, WebSocket).
    *   **REQ-4.4**: Система повинна надсилати сповіщення про наближення дедлайну для збережених або відмічених грантів.

**Характеристика 5: Адміністративна панель**
*   **Опис і пріоритет**: Середній. Забезпечення інструментів для моніторингу та управління системою.
*   **Функціональні вимоги**:
    *   **REQ-5.1**: Адміністратор повинен мати доступ до дашборду для моніторингу стану ключових сервісів (API, PostgreSQL, Redis, Elasticsearch, Celery).
    *   **REQ-5.2**: Система повинна надавати адміністратору інтерфейс для ручного запуску, зупинки та перегляду логів завдань збору даних (ingestion jobs).
    *   **REQ-5.3**: Адміністратор повинен мати можливість переглядати список всіх зареєстрованих користувачів, їхню останню активність та, за необхідності, тимчасово блокувати доступ.
    *   **REQ-5.4**: Система повинна збирати та візуалізувати у вигляді графіків базову аналітику: динаміку реєстрації нових користувачів, кількість пошукових запитів по днях, топ-10 найпопулярніших грантів.
    *   **REQ-5.5**: Адміністратор повинен мати можливість розсилати системні повідомлення всім користувачам.

### 2.3.3. Вимоги зовнішніх інтерфейсів

1.  **Користувацькі інтерфейси (GUI)**: Інтерфейс має бути реалізований як Single Page Application (SPA) з адаптивним дизайном для коректного відображення на десктопних та мобільних пристроях. Всі оновлення даних (стрічка рекомендацій, результати пошуку) повинні відбуватися динамічно, без перезавантаження сторінки.
2.  **Програмні інтерфейси (API)**: Система повинна інтегруватися з такими зовнішніми REST API:
    *   **ORCID API**: для автентифікації та отримання даних профілю.
    *   **Grants.gov API**: для отримання інформації про гранти.
    *   **OpenAlex API**: для збагачення даних про публікації.
3.  **Комунікаційні інтерфейси**: Для сповіщень у реальному часі взаємодія між клієнтом та сервером повинна здійснюватися через протокол **WebSocket**.

### 2.3.4. Інші нефункціональні вимоги

1.  **Продуктивність**: Час відповіді API для пошукових запитів не повинен перевищувати 500 мс при середньому навантаженні. Сторінки повинні завантажуватися не довше 2 секунд. Затримка при передачі WebSocket-повідомлень має бути мінімальною (<100 мс).
2.  **Безпека**: Доступ до захищених маршрутів API повинен контролюватися за допомогою JWT-токенів з коротким терміном дії. Всі чутливі дані (паролі, ключі до API) повинні зберігатися в безпечному сховищі (наприклад, HashiCorp Vault) або через змінні середовища і не повинні бути частиною кодової бази. Система повинна бути захищена від основних веб-вразливостей (XSS, CSRF, SQL-ін'єкції).
3.  **Надійність**: Система повинна бути стійкою до збоїв у зовнішніх API, використовуючи механізми повторних спроб (retry) та таймаутів. Збій одного з джерел даних не повинен призводити до повної відмови системи. База даних повинна мати механізм резервного копіювання.
4.  **Масштабованість**: Архітектура повинна дозволяти горизонтальне масштабування веб-сервера (FastAPI) та фонових воркерів (Celery) для обробки зростаючого навантаження шляхом запуску додаткових контейнерів.
5.  **Сумісність**: Клієнтська частина повинна коректно працювати в останніх версіях популярних браузерів (Chrome, Firefox, Safari, Edge).

# РОЗДІЛ 3. АРХІТЕКТУРА ТА ПРОЄКТУВАННЯ СИСТЕМИ

## 3.1. Проєктування архітектури системи

Для реалізації серверної частини застосунку було обрано підхід **Чистої архітектури (Clean Architecture)**. Ця модель забезпечує максимальне розділення відповідальності (Separation of Concerns), слабку зв'язність компонентів та високий рівень абстракції від зовнішніх фреймворків та інструментів, що значно полегшує тестування, підтримку та подальший розвиток системи.

Серверна частина (бекенд) логічно розділена на чотири концентричні шари, де залежності спрямовані виключно всередину.

1.  **Domain Layer (Шар предметної області)**:
    *   **Призначення**: Це ядро системи, що містить всю бізнес-логіку та правила. Він є повністю незалежним від будь-яких технологій та фреймворків.
    *   **Компоненти**:
        *   **Сутності (Entities)**: Класи, що представляють ключові об'єкти системи (наприклад, `User`, `Opportunity`, `Publication`). У нашому випадку це моделі SQLAlchemy, що містять атрибути та базові методи для валідації стану. Розташовані в `app/domain/entities.py`.
        *   **Винятки (Exceptions)**: Спеціалізовані класи винятків, що описують помилки бізнес-логіки (наприклад, `ProfileNotFound`). Розташовані в `app/domain/exceptions.py`.

2.  **Application Layer (Прикладний шар)**:
    *   **Призначення**: Координує виконання бізнес-логіки, реалізуючи сценарії використання системи (Use Cases). Цей шар не містить логіки роботи з базами даних чи HTTP, а лише оркеструє роботу доменного шару.
    *   **Компоненти**:
        *   **Сценарії використання (Use Cases)**: Класи, що інкапсулюють логіку конкретної операції (наприклад, `GetUserProfile`, `RecommendOpportunities`). Розташовані в `app/application/use_cases/`.
        *   **Порти (Ports)**: Абстрактні класи (інтерфейси), що визначають контракти для взаємодії з інфраструктурним шаром. Прикладами є `OpportunitiesRepository` (для доступу до даних про гранти) або `UnitOfWork` (для управління транзакціями). Розташовані в `app/application/ports/`.

3.  **Infrastructure Layer (Інфраструктурний шар)**:
    *   **Призначення**: Реалізація абстракцій (портів), визначених у прикладному шарі. Цей шар містить весь "брудний" код, що залежить від конкретних технологій.
    *   **Компоненти**:
        *   **Реалізації сховищ (Repositories)**: Класи, що реалізують інтерфейси репозиторіїв для роботи з PostgreSQL через SQLAlchemy (наприклад, `SqlAlchemyOpportunitiesRepository`). Розташовані в `app/infrastructure/persistence/sqlalchemy/`.
        *   **Unit of Work**: Конкретна реалізація патерну `Unit of Work` для управління сесіями та транзакціями SQLAlchemy.
        *   **Зовнішні сервіси (Source Connectors)**: Модулі для взаємодії з зовнішніми API (Grants.gov, OpenAlex, ORCID). Розташовані в `app/integrations/`.

4.  **Presentation Layer (Шар представлення)**:
    *   **Призначення**: Зовнішній шар, що відповідає за взаємодію з клієнтом. У нашому випадку це REST API.
    *   **Компоненти**:
        *   **API Endpoints**: Модулі FastAPI (`admin.py`, `profiles.py` тощо), що визначають маршрути, обробляють HTTP-запити, валідують дані та викликають відповідні Use Cases. Розташовані в `app/api/`.
        *   **DTO (Data Transfer Objects)**: Схеми Pydantic, що використовуються для валідації вхідних даних та серіалізації вихідних. Вони забезпечують чіткий контракт між клієнтом та сервером. Розташовані в `app/schemas/`.

### Архітектура комунікації

Система використовує гібридний підхід до комунікації:
*   **REST API**: Основний механізм взаємодії між клієнтською та серверною частинами. Всі основні CRUD-операції (створення, читання, оновлення, видалення), пошук та отримання рекомендацій реалізовані через синхронні та асинхронні HTTP-запити.
*   **WebSockets**: Використовуються для проактивного сповіщення клієнта про події в реальному часі. Наприклад, коли фонове завдання Celery завершує збір нових грантів і знаходить відповідність для активного користувача, сервер ініціює надсилання повідомлення через WebSocket. Це дозволяє миттєво оновити інтерфейс користувача без необхідності періодичних запитів (polling) з боку клієнта.

## 3.2. Проєктування логічної та фізичної структури бази даних

Для збереження даних системи використовується реляційна СУБД **PostgreSQL**. Структура бази даних спроєктована з урахуванням необхідності зберігати складні зв'язки між дослідниками, їхніми науковими інтересами, публікаціями та грантами. Взаємодія з базою даних здійснюється через моделі SQLAlchemy ORM, визначені в `app/domain/entities.py` та модулях.

Нижче наведено опис ключових таблиць (сутностей) бази даних.

1.  **`users`**: Основна таблиця для зберігання облікових записів.
    *   **Призначення**: Містить базову інформацію для автентифікації та ідентифікації користувачів.
    *   **Ключові поля**: `id` (PK), `email`, `hashed_password` (для адміністраторів), `orcid_id` (для дослідників), `role` (зв'язок з `user_roles`).

2.  **`researcher_profiles` та `researcher_profile_details`**: Таблиці для зберігання розширеної інформації про дослідників.
    *   **Призначення**: Формування детального профілю дослідника, що є основою для рекомендаційної системи.
    *   **Ключові поля**:
        *   `researcher_profiles`: `id` (PK), `user_id` (FK до `users`), `full_name`.
        *   `researcher_profile_details`: `profile_id` (FK до `researcher_profiles`), `bio`, `interests` (масив тегів), `website`.

3.  **`opportunities`**: Таблиця для зберігання інформації про гранти та можливості фінансування.
    *   **Призначення**: Центральна таблиця, що агрегує дані про гранти з різних джерел.
    *   **Ключові поля**: `id` (PK), `title`, `description`, `funding_amount`, `deadline`, `source_url`, `source_id` (FK до `opportunity_sources`).

4.  **`publications`** (неявна, частина профілю): Хоча окремої таблиці `publications` може не бути, дані про публікації, отримані з ORCID, зберігаються у зв'язку з профілем дослідника, ймовірно, у вигляді JSON-поля в `researcher_profile_details` або в окремій пов'язаній таблиці для забезпечення можливості пошуку по них.

5.  **`ingestion_batches` та `ingestion_errors`**: Таблиці для моніторингу процесу збору даних.
    *   **Призначення**: Забезпечення надійності та прозорості роботи підсистеми збору даних.
    *   **Ключові поля**:
        *   `ingestion_batches`: `id` (PK), `source`, `status`, `start_time`, `end_time`, `records_processed`.
        *   `ingestion_errors`: `id` (PK), `batch_id` (FK до `ingestion_batches`), `error_message`, `raw_data`.

6.  **`profile_opportunity_statuses`**: Асоціативна таблиця для відстеження взаємодії користувача з грантами.
    *   **Призначення**: Зберігає статус конкретного гранту для конкретного користувача (наприклад, "збережено", "не цікаво", "подано заявку"). Ця інформація використовується для покращення рекомендацій.
    *   **Ключові поля**: `profile_id` (FK), `opportunity_id` (FK), `status`.

7.  **`notifications` та `opportunity_reminders`**: Таблиці для управління системою сповіщень.
    *   **Призначення**: Зберігання створених сповіщень та нагадувань про дедлайни.
    *   **Ключові поля**: `id` (PK), `user_id` (FK), `content`, `type` (`new_opportunity`, `deadline_reminder`), `status` (`sent`, `read`).

Схема даних спроєктована з використанням нормалізації для уникнення надлишковості даних та забезпечення їхньої цілісності за допомогою зовнішніх ключів та обмежень.

## 3.3. Проєктування алгоритмів та бізнес-логіки

Специфіка розробленого програмного забезпечення вимагає реалізації кількох ключових алгоритмів для обробки даних, генерації рекомендацій та забезпечення ефективного пошуку.

### 3.3.1. Алгоритм збору та нормалізації даних (Ingestion)

Цей алгоритм є основою для наповнення системи даними і працює у вигляді фонового завдання, керованого Celery.

1.  **Ініціація**: Завдання запускається автоматично за розкладом (наприклад, раз на добу) або вручну адміністратором. Створюється запис у таблиці `ingestion_batches` зі статусом `RUNNING`.
2.  **Отримання даних (Extraction)**: Система послідовно звертається до зовнішніх API (Grants.gov, OpenAlex). Дані отримуються посторінково (пагінація) для уникнення перевантаження API та обробки великих обсягів інформації.
3.  **Перетворення (Transformation)**: Кожен отриманий запис проходить етап нормалізації:
    *   Текстові поля (опис гранту) очищуються від HTML-тегів та спеціальних символів.
    *   Дати приводяться до єдиного формату (ISO 8601).
    *   Числові значення (сума фінансування) валідуються та перетворюються у відповідний числовий тип.
4.  **Дедуплікація**: Перед збереженням система перевіряє наявність гранту в базі даних за унікальним ідентифікатором з джерела (`source_id`). Це запобігає створенню дублікатів.
5.  **Завантаження (Loading)**: Очищені та валідовані дані зберігаються в таблиці `opportunities` в рамках однієї транзакції бази даних.
6.  **Індексація**: Після успішного збереження в PostgreSQL, дані про новий грант надсилаються в Elasticsearch для індексації та забезпечення доступності для повнотекстового пошуку.
7.  **Завершення та логування**: Після завершення обробки всіх даних статус у `ingestion_batches` оновлюється на `COMPLETED`. У разі виникнення помилок під час обробки запису, він пропускається, а інформація про помилку (включаючи вихідні "сирі" дані) зберігається в таблиці `ingestion_errors` для подальшого аналізу адміністратором.

### 3.3.2. Алгоритм персоналізованих рекомендацій

Це ядро інтелектуальної складової системи. Алгоритм працює на основі контент-орієнтованого підходу (content-based filtering).

1.  **Побудова вектору профілю користувача**:
    *   Система агрегує всі текстові дані, пов'язані з дослідником: анотації його наукових публікацій, ключові інтереси, вказані в профілі, та біографію.
    *   Цей сукупний текст обробляється: видаляються стоп-слова, застосовується лематизація (приведення слів до базової форми).
    *   За допомогою попередньо навченої моделі (наприклад, **Doc2Vec** або **TF-IDF**, розрахованого на корпусі наукових текстів) оброблений текст перетворюється на числовий вектор фіксованої довжини — **вектор профілю**.

2.  **Побудова векторів грантів**:
    *   Аналогічна процедура виконується для кожного гранту в системі. Його опис та назва перетворюються на числовий **вектор гранту**. Цей процес виконується один раз при додаванні нового гранту до системи.

3.  **Розрахунок релевантності**:
    *   Коли користувач запитує рекомендації, система обчислює **косинусну подібність (cosine similarity)** між вектором профілю користувача та векторами всіх доступних грантів.
    *   Косинусна подібність є мірою, що показує, наскільки два вектори спрямовані в одному напрямку (значення від -1 до 1). Значення, близьке до 1, означає високу семантичну близькість між інтересами дослідника та тематикою гранту.

4.  **Ранжування та фільтрація**:
    *   Гранти ранжуються за спаданням показника косинусної подібності.
    *   Зі списку виключаються гранти, які користувач раніше позначив як "нецікаві", а також ті, у яких вийшов термін подачі.
    *   Кінцевий відсортований список повертається користувачеві.

### 3.3.3. Алгоритм повнотекстового пошуку

Для реалізації швидкого та гнучкого пошуку використовується рушій **Elasticsearch**.

1.  **Індексація**: Як було зазначено, всі дані про гранти індексуються в Elasticsearch. Під час індексації до текстових полів застосовуються спеціальні аналізатори, які виконують токенізацію, перетворення в нижній регістр та морфологічний аналіз (stemming), що дозволяє шукати за різними формами одного слова.
2.  **Обробка запиту**: Коли користувач вводить пошуковий запит, він надсилається на бекенд.
3.  **Формування запиту до Elasticsearch**: Бекенд формує складний JSON-запит до Elasticsearch API, який може включати:
    *   `multi_match`: для пошуку введеного тексту одразу в кількох полях (наприклад, `title` та `description`).
    *   `bool` з секціями `must` та `filter`: для комбінування повнотекстового пошуку з точними фільтрами (наприклад, діапазон дат, сума фінансування).
    *   `function_score`: для додаткового підвищення рейтингу документів, які, наприклад, мають більш свіжу дату.
4.  **Отримання та відображення результатів**: Elasticsearch повертає відсортований за релевантністю список документів, який потім передається на фронтенд для відображення. Завдяки індексації цей процес займає мілісекунди навіть на великих обсягах даних.
