# MVP to V1 Enhancement Plan

This plan turns the current working MVP into a more professional V1 product and a stronger academic/demo project. It combines the selected engineering enhancements, the current roadmap, and the ideas from `ideas.txt`.

## Current Baseline

The MVP already includes:

- FastAPI backend with SQLAlchemy, Alembic, auth, user-owned profiles, and tests.
- React + TypeScript frontend dashboard.
- Opportunity catalog, filters, recommendation feed, saved/planned/applied statuses.
- Grants.gov ingestion, generic RSS/JSON external source imports, ORCID, and OpenAlex enrichment.
- RQ/Redis workers and scheduler.
- Local deterministic semantic embeddings.
- Notification preferences/history and reminder notification records.
- Admin dashboard, audit log, source batches, duplicate merge, analytics.
- Application assistant that generates checklist, motivation outline, fit statement, warnings, and export notes.
- Makefile shortcuts for setup, run, test, migration, and Docker.

## V1 Goals

V1 should feel like an intelligent research career assistant, not only an opportunity search tool.

Product goals:

- Improve matching quality and make the scoring intelligence visible.
- Use production-like semantic search with local embeddings and pgvector.
- Add real notification delivery.
- Make operations/admin workflows reliable.
- Improve source coverage with specific connectors.
- Make the UI smoother, more professional, and easier to trust.
- Add academic/demo features that clearly show algorithmic depth.

Non-goal for now:

- CI/CD and hosted deployment. Leave this for the final infrastructure phase.

## Milestone 1: Production-Grade Semantic Matching

### 1.1 Local Model Embeddings

Replace deterministic hash embeddings with local model embeddings.

Recommended approach:

- Add an embedding provider abstraction:
  - `local_stub` for tests;
  - `sentence_transformers` or another local model for dev/prod;
  - optional future provider for OpenAI or another external API.
- Store embedding model name, dimensions, and generated timestamp.
- Regenerate embeddings through worker jobs.

Acceptance criteria:

- Profile and opportunity embeddings are generated by a real local model.
- Tests can run without downloading models by using a deterministic test provider.
- Embedding refresh jobs support batch processing.
- README documents model setup and fallback mode.

Status:

- Embedding provider abstraction added.
- `hash` provider remains the deterministic test/local fallback.
- Optional `sentence-transformers` provider is configurable through `EMBEDDING_PROVIDER=sentence_transformers`.
- Embedding model name is persisted with generated embeddings.

### 1.2 pgvector Similarity Queries

Move semantic search from Python-side cosine calculation to PostgreSQL + pgvector.

Tasks:

- Add real `vector` columns for PostgreSQL.
- Keep SQLite-compatible text fallback for local lightweight mode if needed.
- Add pgvector indexes.
- Query top semantic candidates before deterministic reranking.
- Keep Alembic migrations safe for SQLite and PostgreSQL.

Acceptance criteria:

- Docker/PostgreSQL mode uses pgvector similarity.
- Recommendation endpoint does not load every opportunity into memory.
- Semantic candidate count is configurable.

Status:

- PostgreSQL migration adds pgvector columns and an ivfflat cosine index.
- Recommendation endpoint uses pgvector candidate retrieval in PostgreSQL mode.
- SQLite keeps the portable Python fallback.

### 1.3 Hybrid Ranking Formula

Make ranking more explicit and tunable:

```text
final_score =
  w_semantic * semantic_score +
  w_eligibility * eligibility_score +
  w_deadline * deadline_score +
  w_history * user_history_score +
  w_quality * source_quality_score
```

Tasks:

- Split current score into components.
- Return score breakdown in recommendation responses.
- Add admin/user-visible explanation of why the result ranked where it did.
- Store ranking configuration in settings.

Acceptance criteria:

- UI shows score breakdown, not only one match score.
- Tests cover component-level scoring.
- README gets an “Algorithm Description” section.

Status:

- Recommendation response now returns semantic, eligibility, deadline, user history, and final score components.
- Frontend opportunity cards show the score breakdown.
- Ranking weights are configurable.

### 1.4 Feedback Learning

Use saved/ignored/applied history to adapt rankings.

Ideas from `ideas.txt`:

- If a user repeatedly ignores a country/topic/source, reduce its weight.
- If a user saves or applies to a type/source/topic, raise its weight.
- Make the behavior explainable, not mysterious.

Tasks:

- Add user preference signals derived from statuses.
- Create profile-level learned preferences table or computed service.
- Add “because you saved similar…” and “ranked lower because you ignored…” explanations.

Acceptance criteria:

- Recommendation order changes based on user history.
- Ignored patterns affect future rankings.
- Explanations identify history-based effects.

Status:

- Saved/planned/applied/accepted and ignored statuses are converted into history signals.
- History signals adjust topic, country, and opportunity-type ranking.
- Explanations include “Ranks higher/lower…” history reasons.

## Milestone 2: Requirement Extraction and Eligibility Intelligence

### 2.1 LLM/Parser Requirement Extraction

Grant texts are often messy. Add a parser that extracts structured requirements.

Extract:

- required career stage;
- eligible countries/regions;
- required degree;
- publication expectations;
- mobility constraints;
- citizenship/residency constraints;
- age/year-since-PhD constraints where present;
- ethical or compliance requirements.

Implementation path:

- Start with deterministic regex/rule extraction.
- Add optional LLM/local model extraction later.
- Store extracted requirements separately from raw eligibility text.
- Add confidence and source snippet fields.

Acceptance criteria:

- Opportunity detail page shows parsed requirements.
- Recommendation scoring uses parsed requirements when available.
- Application assistant warnings use parsed requirements.

Status:

- Deterministic requirement extractor added for career stage, country/region, degree, language, publication, mobility, citizenship, and years-since-PhD signals.
- Extracted requirements are stored on opportunities with confidence and exposed in opportunity API responses.
- Recommendation scoring uses parsed career-stage/country requirements when explicit metadata is sparse.
- Frontend opportunity details show parsed requirements in the eligibility tab.

### 2.2 Success Probability and Gap Analysis

Move beyond “match score” into “what is missing.”

Ideas from `ideas.txt`:

- Show where the researcher is strong and weak.
- Example: strong topic fit, but missing recent publications in a target field.
- Provide action steps.

Tasks:

- Add `success_probability` or `readiness_score`.
- Add gap categories:
  - topic fit;
  - career stage;
  - country/mobility;
  - publications;
  - language;
  - funding history;
  - deadline readiness.
- Add UI panel: “How to improve your chances.”

Acceptance criteria:

- Opportunity detail explains gaps clearly.
- Assistant export includes gap analysis.
- Tests cover at least three gap scenarios.

Status:

- Application readiness score and gap analysis are generated for each opportunity/profile pair.
- Recommendations now include readiness score, strengths, and gaps.
- Application Assistant output and export notes include readiness, strengths, and gaps.
- Application Assistant now keeps deterministic facts as the source of truth and can optionally use Groq or a local OpenAI-compatible model to generate a polished advisor memo.
- Tests cover parser extraction, sparse-metadata recommendation scoring, and assistant gap analysis.

## Milestone 3: Real Notifications

### 3.1 Email Delivery Provider

Replace logged “sent” notifications with real email delivery.

Recommended approach:

- Add email provider abstraction:
  - `console` provider for local/dev;
  - SMTP provider;
  - optional SendGrid/Mailgun/Resend provider.
- Store delivery provider message id.
- Record delivery attempts and failures.

Acceptance criteria:

- Local mode prints email content.
- Production config can use SMTP or provider API.
- Notification status reflects delivery success/failure.

Status:

- Email provider abstraction added.
- `console` provider is the default for local/dev.
- SMTP provider is available through environment configuration.
- Notifications persist recipient, provider, provider message id, delivery attempts, and last error.
- Delivery failures keep notifications pending and record the provider error.

### 3.2 Weekly Digest and High-Match Alerts

Use stored preferences from Phase 8.

Tasks:

- Worker job for weekly recommendation digest.
- Worker job for new high-match opportunity alerts.
- Digest includes top matches, deadlines, and why each matters.
- Deduplicate alerts so the same opportunity is not sent repeatedly.

Acceptance criteria:

- Users can enable/disable each notification type.
- Notification history shows digest/alert records.
- Worker tests cover enabled, disabled, and no-matches cases.

Status:

- Weekly digest worker job added.
- High-match alert worker job added.
- Scheduler enqueues digest and alert jobs based on configurable intervals.
- Manual job endpoints added for digest and high-match alerts.
- Frontend admin operations can queue digest and alert jobs.
- Tests cover digest creation, high-match alert creation, and delivery failure recording.

## Milestone 4: Admin Security and API Professionalization

### 4.1 Role Enforcement

Admin endpoints must require admin users.

Tasks:

- Add `require_admin_user` dependency.
- Protect:
  - `/admin/*`;
  - bulk imports if desired;
  - source diagnostics;
  - duplicate merge;
  - job retry endpoints if needed.
- Add tests for researcher vs admin access.

Acceptance criteria:

- Researcher users get `403` for admin APIs.
- Admin users can access admin APIs.
- Legacy MVP anonymous paths are intentionally documented or removed.

Status:

- `/admin/*` routes now require authenticated admin users.
- `/jobs/*` queue visibility/enqueue/retry routes now require authenticated admin users.
- Frontend admin/job calls now send the signed-in token.
- Tests cover researcher denial and admin access.

### 4.2 Request IDs and Error Envelopes

Add professional API behavior.

Tasks:

- Generate request id per request.
- Return `X-Request-ID` response header.
- Standardize error body:

```json
{
  "error": {
    "code": "not_found",
    "message": "Opportunity not found",
    "request_id": "..."
  }
}
```

- Log request id with exceptions.

Acceptance criteria:

- All handled HTTP errors use the envelope.
- Unexpected exceptions include request id.
- Tests verify response shape and header.

Status:

- Request middleware now assigns or echoes `X-Request-ID` and logs it.
- HTTP and validation errors now return a standard `{ "error": { "code", "message", "request_id" } }` envelope.
- Tests cover request id propagation, not found errors, validation errors, and admin forbidden errors.

### 4.3 Rate Limiting and External API Safety

From `ideas.txt`: ORCID/OpenAlex and source imports need rate limiting.

Tasks:

- Add per-route rate limits for ingestion/integrations.
- Add backoff and retry policy for external APIs.
- Cache OpenAlex/ORCID responses where safe.
- Respect provider etiquette.

Acceptance criteria:

- Burst requests are rejected or queued.
- External connector tests cover retry/backoff behavior.

  - source;
  - date/deadline;

Acceptance criteria:

Status:

## Product Questions To Discuss

These came from light manual V1 testing and need product decisions before deeper implementation:

- Account/profile model: decide whether users should have multiple research profiles, or whether the active profile switcher should be removed for a simpler one-account-one-profile flow.
- Status taxonomy: define exact user-facing meaning for saved, planned, applied, accepted, rejected, and ignored, then decide whether saved/planned should become more explicit labels like "saved" and "in application plan."
- Reminder lifecycle: decide whether completed reminders should be hidden by default, archived in a separate view, or replaced with a snooze/dismiss model.
- Application Assistant depth: current output is structured but still mostly deterministic. Decide whether V1 should add requirement extraction, opportunity-specific document prompts, CV tailoring, or an optional LLM/local model provider before calling it complete.
- Controlled vocabularies: decide whether country, discipline, source, and keyword fields should use strict dropdown taxonomies or continue with datalist hints from the current opportunity database.
